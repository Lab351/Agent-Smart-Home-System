# Room Agent 事件路由与 LLM 分层执行架构

本文档对“事件进入 -> 规则引擎 -> LLM 路由 -> 执行”的架构图进行文字化说明，便于后续在 LangGraph 中落地实现。

## 1. 架构目标

该架构的核心目标是：

- 将确定性逻辑与生成式推理解耦。
- 在保证响应质量的前提下，优先走低成本路径。
- 对工具调用和高风险动作增加可审计、可插入人工审批的控制点。

## 2. 输入事件与前置约束

系统接收两类事件：

- 传感器事件：来自设备侧状态变化。
- Agent 事件：来自上层智能体或用户请求。

图中前置注记的含义是：如果事件类型存在明显异构（频率、时序、可靠性差异较大），需要额外的设备状态时序存储，或者在下层先做归一化处理，再进入统一决策流。

## 3. 第一层：规则引擎（确定性路径）

规则引擎区域负责快速、稳定、可解释的处理，主要由两部分组成：

- 触发推理：判断当前事件是否满足某条策略的触发条件。
- 自动化引擎：执行预定义规则编排（不依赖 LLM）。

这一路径对应“高确定性、低延迟”场景。若规则已能完整覆盖，直接执行，不进入昂贵推理流程。

## 4. LLM 可用性判定与分流

当规则路径不能直接闭环时，进入“LLM 可用？”判定节点。

- N 分支（不可用或不建议使用）：进入优雅降级路径。
- Y 分支（可用）：进入完整 LLM 工作流。

该判定可综合以下因素：

- 模型服务健康度（可达性、超时、限流）。
- 成本预算与配额。
- 当前请求复杂度与时效要求。
- 隐私策略或工具权限约束。

## 5. 第二层：低成本降级路径

在 LLM 不可用时，系统进入轻量流程：

- 传统意图识别：使用规则模板、关键词或轻量分类器。
- 执行：在限定能力范围内完成任务。

这条路径保证系统始终可用，避免因外部模型波动导致全链路阻塞。

## 6. 第三层：LLM 工作流（高能力路径）

LLM 工作流对应复杂任务处理，包含以下阶段：

1. 意图识别：由低成本模型或主模型做语义理解。
2. 工具判定：识别是否需要外部工具/系统调用。
3. 直接回复或工具编排：
   - 不需要工具：直接回复。
   - 需要工具：进入工具选择和工具执行。
4. 工具 / 文本推理输出：产出结构化结果或自然语言结果。
5. 可选人工审计：在高风险动作前加入人工确认。
6. 执行：落地到设备控制、状态更新或外部系统回写。

## 7. 颜色语义（与图例一致）

- 蓝色：确定性流程（规则和可预测执行）。
- 绿色：低成本 LLM 或轻推理节点。
- 粉色：较强 LLM 与工具输出阶段（成本更高、能力更强）。

## 8. 设计收益

该分层结构可带来以下收益：

- 成本可控：默认优先规则和轻量路径。
- 可靠性更高：模型不可用时可降级，不中断主流程。
- 安全可审计：高风险动作支持人工审计插点。
- 便于工程化：每个节点可独立测试、观测与替换。

## 9. 对 room-agent 的实现建议

在 room-agent 中可将该图映射为如下模块边界：

- 规则层：事件归一化、策略匹配、自动化执行节点。
- 路由层：LLM 健康判定、成本策略判定、分支路由。
- LLM 层：意图识别、工具选择、工具调用编排。
- 治理层：人工审批钩子、审计日志、失败回退。
- 执行层：设备控制与状态落库。

建议在 LangGraph 中把“规则路径”和“LLM 路径”设计为两个可独立演进的子图，并在共享状态中统一记录：`intent`、`selected_tools`、`risk_level`、`approval_required`、`execution_result`。
